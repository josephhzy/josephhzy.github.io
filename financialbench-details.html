The Problem

LLMs hallucinate financial data with confidence. Ask a model for DBS Group's CET1 ratio or Singtel's revenue, and it will give you a precise-sounding number -- often wrong, sometimes dangerously close to right. I wanted to measure this properly: where do they fail, how often, and is the failure pattern predictable.

Anecdotal testing is not enough -- you need ground truth, controlled experiments across prompt strategies and temperature settings, and statistical rigour to distinguish random variance from systematic error. So I built a framework to generate that evidence.

The Approach

Standard LLM benchmarks test general knowledge and reasoning. They do not answer the question that matters for financial applications: can this model reliably reproduce a specific number from a specific report, and will it give the same answer if asked differently? This framework tests that directly, on real financial data from SGX-listed companies.

The ground-truth set contains 33 manually verified numerical facts across five SGX-listed companies, chosen to cover the kinds of figures that appear in real financial workflows: capital ratios, revenue, profitability, and balance-sheet disclosures. Small but high-precision — every fact is sourced from published annual reports with exact page references.

How It Works

  • Runs 660 structured evaluation runs against a ground-truth dataset of verified financial facts from SGX-listed companies (DBS, OCBC, UOB, Singtel, CapitaLand)
  • Measures response quality across three dimensions: semantic consistency, factual consistency, and hallucination rate
  • Supports multi-model comparison -- test the same fact set across Anthropic Claude, OpenAI GPT, and other providers to surface model-level differences in reliability
  • Uses an independent extraction stage to eliminate evaluator bias -- the model under test never scores its own outputs
  • Tests across multiple prompt templates and temperature levels with repeated runs per combination to measure variance
  • Scores each fact with a composite stability score and traffic-light flag (green / yellow / red)
  • Deployed on Streamlit Cloud with interactive dashboards for per-fact, per-model, and per-template breakdowns

What This Tells You Before Deployment

  • Hallucination rates, measured — not anecdotal, but quantified across hundreds of controlled runs so you know the baseline failure rate
  • Random variance vs systematic error — the model gives different wrong answers each time (variance) or the same wrong answer consistently (bias). Different failure modes need different mitigations
  • Model comparison for selection decisions — objective data on which model is more reliable on domain-specific numerical facts, tested under identical conditions
  • Where human verification is most needed — large absolute figures hallucinate more than percentages, so you know which outputs to double-check

Model Comparison

The framework supports head-to-head model comparison on the same fact set. This enables objective, data-driven model selection rather than relying on general-purpose benchmarks that may not reflect performance on domain-specific financial data. Each model is tested under identical conditions -- same facts, same prompts, same temperature settings -- so differences in reliability are attributable to the model, not the evaluation setup.

01 Prompt wording matters more than temperature

The best-performing prompt template scored 0.978 stability while the worst scored 0.434. That is more than double the reliability from the same model on the same facts, purely from prompt design. That means reliability is partly a system design problem, not just a model selection problem — prompt engineering is a primary control lever.

02 LLMs handle percentages better than large numbers

Percentage-based metrics (Net Interest Margin, Return on Equity) were consistently the most stable facts. Large absolute figures in the hundreds of billions were the least stable. The model struggles most with precisely the data types that matter most in financial analysis -- a finding that directly informs where human verification is most needed.

03 Lower temperature helps, but does not solve it

T=0.0 scored 0.723 vs T=1.0 at 0.530. Setting temperature to zero makes the model more deterministic -- but it still hallucinates the same wrong answer consistently. Determinism is not the same as correctness. This is a critical distinction for systems that rely on temperature=0 as a reliability measure.

04 Evaluation integrity requires unit normalisation

65% of responses were flagged as hallucinations -- but only after unit normalisation was applied in the scoring layer. Without it, a model answering "S$3 billion" would be incorrectly flagged because the ground truth is stored as 3000 (SGD millions). This finding demonstrates that evaluation methodology matters -- a poorly designed evaluator produces misleading results.

Key Features

  • 660 structured evaluation runs covering verified financial facts from 5 SGX-listed companies
  • Multi-model comparison -- provider-agnostic adapter pattern for testing across Anthropic, OpenAI, and other providers
  • Interactive Streamlit dashboard deployed on Streamlit Cloud for exploring results by model, company, template, and temperature
  • Checkpointing every 50 calls -- runs are resumable after crashes without losing progress
  • Cost controls -- estimated cost is shown and confirmed before expensive runs execute
  • YAML-based configuration -- all evaluation parameters externalised for reproducible experimentation
  • Structured CSV reporting -- per-fact, by-company, by-template, and by-temperature exports

What I Learned

A few things I ran into while building this:

  • Floating-point arithmetic is treacherous in scoring. Comparing extracted values like "22,297 million" against ground truth requires careful handling. 22297.0 / 1000000 * 1000000 can give 22296.999999 in Python. I used explicit rounding with defined precision and deterministic tie-breaking for modal values.
  • Make optional dependencies truly optional. The semantic consistency metric requires sentence-transformers (a heavy GPU library). In CI or lightweight environments, the scorer should gracefully return a default value with a warning -- not crash the entire evaluation pipeline.
  • Log what the model actually said, not just that it failed. When the extractor cannot parse a number from an LLM response, knowing the raw response text is critical for debugging. "Extraction failed" tells you nothing; "Extraction failed on: 'Revenue was approximately twenty-two billion'" tells you the regex needs to handle word-form numbers.
  • 5% tolerance is not arbitrary. Financial reports present numbers with rounding ("~S$22.3B"). The tolerance must accommodate legitimate rounding while catching real errors. Testing at 3%, 5%, and 10% showed 5% is the sweet spot for this dataset.

Limitations

  • Ground truth covers 33 facts from 5 SGX-listed companies — a larger fact set would give more statistical power to per-company and per-metric breakdowns
  • Evaluation uses the model's own extraction to parse answers — extraction errors can be misclassified as hallucinations
  • Only tested on Anthropic Claude models so far — multi-provider comparison (GPT-4, Gemini) is supported but not yet run
  • No temporal evaluation — the same facts should be re-tested as models are updated to track reliability drift over time
