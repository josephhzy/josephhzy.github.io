The Problem

Banks make thousands of lending decisions daily. A wrong approval costs the loan amount (often $50K+); a wrong decline costs lost revenue and customer trust. In regulated lending, the model also needs to be explainable to auditors, fair across demographics, and promoted through a governed process. I wanted to work through all of that, not just the modelling.

Why This Was Hard

  • Default is rare. Only 8% of rows are positive — accuracy is meaningless, and naive models learn to predict "no default" for everything
  • Credit decisions need explanation. A score alone is not enough — regulators and customers can ask why they were declined
  • A better metric is not enough to deploy. Model promotion needs governance because performance on a test set does not guarantee fairness or calibration in production

The Approach

Built on a 7-table banking dataset with 307K rows and an 8% default rate. Most of the work ended up being threshold tuning under class imbalance, SHAP explanations, subgroup fairness checks, and setting up the promotion gates.

Data Engineering

The raw dataset spans 7 relational tables representing different aspects of banking customer records: account information, transaction history, credit bureau data, payment patterns, and demographic attributes. The engineering pipeline joins these tables, handles missing values with domain-appropriate strategies (not just mean imputation), and constructs 87+ features including:

  • Financial ratios -- debt-to-income, credit utilisation, payment-to-balance ratios
  • Temporal aggregations -- rolling averages, trend indicators, payment velocity
  • Behavioural signals -- payment regularity, balance volatility, credit line usage patterns
  • Bureau-derived features -- inquiry counts, delinquency history, account age metrics

Imbalanced Classification

Credit default is a rare event -- the dataset exhibits severe class imbalance typical of real banking portfolios. The system addresses this through multiple complementary strategies:

  • SMOTE oversampling applied within cross-validation folds to prevent data leakage
  • Threshold tuning based on precision-recall tradeoffs rather than defaulting to 0.5
  • Stratified cross-validation ensuring each fold preserves the class distribution
  • Class-weighted loss functions in gradient boosting models

Explainability and Fairness

Regulated credit decisioning requires more than a prediction score. CreditGuard provides:

Explainability
SHAP-based interpretability

Every prediction comes with SHAP values explaining which features drove the decision and by how much. Global feature importance, local per-prediction explanations, and interaction effects are all computed and available via the API. This supports regulatory requirements for adverse action explanations in credit decisions.

Fairness
Fairness diagnostics

Fairness checks run before model promotion to ensure error rates and calibration do not degrade across subgroups. Metrics include adverse impact ratio, equalised odds, and calibration by subgroup. A material disparity on any dimension blocks promotion — fairness is a gate, not a report.

Champion-Challenger Governance

Model promotion from development to production follows a governed champion-challenger process tracked through MLflow. A new model (challenger) must demonstrate statistically significant improvement over the current production model (champion) on held-out data before it can be promoted. All experiments, hyperparameters, metrics, and artifacts are versioned and logged, providing a complete audit trail for model governance reviews.

What I Learned

A few things I ran into while building this:

  • Never swallow exceptions silently. My initial SHAP computation had a bare except Exception that silently fell back to feature importances. A real bug in the SHAP integration was hidden for weeks. I learned to catch specific exception types and always log the reason for a fallback.
  • Hyperparameter search is easy to corrupt. When an Optuna trial fails, returning 0.0 (a valid but terrible score) misleads the Bayesian optimizer. The fix was to raise TrialPruned so Optuna treats it as invalid rather than informative. Small difference in code, large difference in search quality.
  • Feature alignment is a silent killer. If the model lacks a feature_names_in_ attribute, it receives features in whatever order the caller sends them. For tree-based models, wrong column order means wrong predictions -- silently. I added a schema file as a fallback alignment source.
  • Validate config at startup, not at request time. Overlapping decision thresholds (approve > decline) would serve wrong decisions for every request. Now validated once at boot.
  • Fairness is not a checkbox. Running demographic_parity() on a public dataset does not prove production fairness. But I still wanted to build the infrastructure -- adverse impact ratio as a promotion gate, proxy sensitivity analysis, calibration by subgroup -- because it forced me to think through the problem properly, even if the dataset is limited.

Limitations

  • Uses the Home Credit public dataset, not live banking data — feature distributions may differ from production portfolios
  • Fairness diagnostics run on proxy attributes, not legally protected classes — real deployment needs HR/legal alignment on sensitive features
  • No online monitoring or drift detection yet — the system evaluates at training time but does not track model decay in production
  • Champion-challenger comparison is offline — there is no live traffic splitting or shadow scoring
